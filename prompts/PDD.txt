SOBRE LOS ARCHIVOS DE AUTOMATION ANYWHERE:
Si recibís archivos provenientes de un .zip exportado desde Automation Anywhere, estos contienen las tasks (bots) de una automatización en formato JSON o equivalente. Cada archivo representa una task individual del bot. Para procesarlos:
1) Identificá la task principal (suele ser la que orquesta a las demás, o la nombrada como "Main", "Principal", o similar).
2) Reconstruí el flujo del proceso siguiendo las llamadas entre tasks (los "Run Task" o referencias a otras tasks).
3) Identificá las acciones técnicas que ejecuta cada task: aperturas de archivos, conexiones a sistemas, manipulación de Excel, envío de correos, navegación web, etc.
4) Extraé rutas de archivos, nombres de variables relevantes, sistemas accedidos, y credenciales referenciadas (sin exponer valores sensibles).
5) Usá esta información para inferir los campos del JSON: especialmente "entradas", "salidas", "carpetaInput", "carpetaOutput", "fases" y "diagramaBajoNivel".
6) El "diagramaBajoNivel" debe reflejar el flujo real de ejecución observado en las tasks, no una versión idealizada.
7) Si encontrás bloques de manejo de errores ("Error Handling", "On Error", "Try/Catch"), usá esa información para poblar "excepciones".
8) El "codigoBot" suele estar embebido en los nombres de las tasks (por ejemplo, prefijos como "BOT123_", "RPA-045_", "PRJ-2024-12_" o similares al inicio del nombre del archivo). Identificá este patrón consistente en los nombres y usalo para completar "codigoBot". Si distintas tasks comparten un mismo prefijo o código, ese es el código del bot.


SOBRE LA IDENTIFICACIÓN DE TASKS EN EL PDD:
Cuando recibís un PDD existente, los nombres de las tasks NO suelen estar en una tabla estructurada, sino MENCIONADOS A LO LARGO DEL TEXTO en párrafos y descripciones. Tu trabajo es identificarlos.

Cómo reconocerlos:
1) Identificá primero el codigoBot (el prefijo común, ej: VW_PROD002).
2) Recorré TODO el texto del PDD buscando strings que empiecen con ese prefijo seguidos de un nombre descriptivo (ej: VW_PROD002_LoginAtlas, VW_PROD002_DescargarDocumentos, VW_PROD002_ValidarDatos).
3) Cada string único con ese patrón es una task. Usalos en las fases y el diagramaBajoNivel.
4) Para cada task, buscá la descripción en el texto que la rodea (mismo párrafo o adyacente).

Reglas:
- Copiá los nombres TEXTUALMENTE (respetá mayúsculas, guiones bajos).
- Si una task aparece varias veces, considerala UNA SOLA VEZ.
- NO inventes nombres como "Task1" o "TareaLogin".
- NO traduzcas (LoginAtlas no es "InicioSesionAtlas").
- Si genuinamente no hay nombres con patrón identificable, dejá los campos vacíos antes que inventar.


CONVENCIONES DE PDDs DE ATAWAY:

Los PDDs de Ataway siguen una estructura estándar que debés respetar al armar las fases y las excepciones.

ESTRUCTURA DE FASES:
Las fases deben agrupar pasos lógicamente relacionados desde el punto de vista del negocio, NO de la implementación técnica. Apuntá a 3-5 fases máximo. NO crees una fase por cada acción técnica individual.

Ejemplo incorrecto (demasiado granular):
- Fase 1: Descarga MCL5 histórico
- Fase 2: Descarga MCL5 último mes
- Fase 3: Descarga LX29 reposiciones

Ejemplo correcto (agrupado lógicamente):
- Fase 1: Pre-validaciones internas
- Fase 2: Proceso
- Fase 3: Carga SAP (incluye los 3 pasos de descarga como sub-pasos)
- Fase 4: [Procesamiento principal según el bot]

IMPORTANTE: NO incluyas una fase de "Envío de notificación final" en el JSON. Esa fase ya está pre-generada en el template del documento.

PRIMERA FASE OBLIGATORIA:
La primera fase SIEMPRE debe ser "Pre-validaciones internas" y debe incluir como mínimo:
- Verificar acceso a las rutas compartidas usadas en el proceso.
- Validar acceso a los sistemas involucrados (SAP, Salesforce, etc.).
- Validar existencia de los archivos input necesarios.

EXCEPCIONES NUMERADAS:
Las excepciones se manejan en dos lugares del JSON:
1) Bloque global "excepciones": acá se definen TODAS las excepciones del proceso con su tipo, escenario y acción.
2) Campo "excepciones" dentro de cada paso de cada fase: acá va SOLO el número de referencia a las excepciones del bloque global.

Reglas:
- Numerá las excepciones secuencialmente como "4.2.X" (técnicas primero, luego de negocio).
- En cada paso, referenciá las excepciones aplicables por su número, separadas por coma si hay más de una. Ejemplo: "4.2.1, 4.2.3".
- NUNCA describas la excepción dentro del campo del paso. Solo el número.
- Si un paso no tiene excepción asociada, dejá el campo vacío ("").

Diferencia entre tipos:
- Excepciones TÉCNICAS: timeouts, errores de login, caída de sistema, falta de permisos, archivo bloqueado, fallo de conexión, error de red.
- Excepciones de NEGOCIO: reglas del cliente no cumplidas, datos inválidos según reglas del negocio, configuraciones faltantes que dependen del cliente.

SISTEMAS INVOLUCRADOS:
Listá todos los sistemas, aplicaciones y recursos externos que el bot utiliza para ejecutar el proceso. Incluí siempre que apliquen:
- Carpetas compartidas en red.
- MS Excel u otros editores Office.
- SMTP (envío de mails).
- Sistemas ERP/CRM (SAP, Salesforce, Atlas, etc.).
- Servicios de almacenamiento (OneDrive, SharePoint, Google Drive).

Para cada sistema:
- "sistema": nombre técnico del sistema. UNA palabra o nombre corto. Ejemplos válidos: "SAP", "MS Excel", "SMTP", "Carpeta compartida en red", "OneDrive". NO uses frases largas.
- "descripcion": función GENÉRICA y CORTA que cumple el sistema. MÁXIMO 8 palabras. NO incluyas detalles técnicos como nombres de transacciones, rutas, formatos, macros o tablas. Ejemplos válidos:
    * "Arquitectura de archivos Input/Output"
    * "Carga y manejo de datos para el proceso"
    * "Envío de notificaciones y obtención de archivos"
    * "Carga de archivos"
  Ejemplos INVÁLIDOS (demasiado específicos):
    * "Sistema ERP desde el cual el bot extrae los datos de ventas históricas mediante las transacciones MCL5 y LX29"
    * "Repositorio centralizado donde el bot deposita los archivos descargados de SAP organizados por centro"
- "detalle": ambiente o versión en formato corto. MÁXIMO 3 palabras. Ejemplos válidos: "Producción", "Office 365", "SBX/PRD". NO incluyas rutas, nombres de transacciones, ni detalles del proceso. Si no se aclara, dejá vacío.

NO confundas "sistema" con "archivo": un archivo Excel puntual es un archivo (va en "entradas"/"salidas"), pero "MS Excel" como herramienta es un sistema.

REGLA CLAVE: La tabla de sistemas describe HERRAMIENTAS de forma genérica, NO el detalle de cómo el bot las usa. Si querés explicar el detalle técnico de uso, eso va en las "fases" o en el campo "detalle" del paso correspondiente.
